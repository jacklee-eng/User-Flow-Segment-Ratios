신호 소스 가이드 (Explicit / Implicit / Cross-sell)

Explicit: 맞춤정보에서 유저가 직접 응답한 데이터 (이사/시공/결혼/출산).

Implicit: 맞춤정보 미입력이어도 행동 시그널로 이사/구매 의향을 추정한 데이터 (Implicit Grade 1).

Cross-sell: 시공/이사/인터넷 등 서비스 상담·계약 이력을 기반으로 교차전환하는 타겟.

| 데이터 소스 | `log.analyst_log_table` (오늘의집 KR 봇 트래픽이 제외된 분석용 로그 테이블) |
| 모수 규모 | 1,380,267명 (2026-05-03 ~ 2026-05-09 7일 unique) |

**모수 규모 검증 결과** (2026-05-09 기준):
- 1일 KR 전체 로그인 UU (앱+웹 합산): 약 39.8만명
- 1일 홈 진입 로그인 UU: 약 41.7만명
- 7일 unique: 1,380,267명 = 1일 평균 약 41만 × 7일 중복 제거
- 일주일 동안 동일 유저가 평균 약 2배 중복 방문 (재방문률 약 48%)
- 결론: **오늘의집 KR 활성 유저 대부분이 홈 화면에 진입함**

### 1.2 카테고리별 데이터 소스

**모수 기간(7일)과 각 카테고리의 분류 기준 기간은 별도로 작동합니다.**
**즉, "최근 7일에 홈 진입한 유저" 모수 안에서, 각 카테고리 조건에 해당하는 유저를 찾아 분류하는 방식입니다.**

| 카테고리 | 데이터 소스 | 식별 조건 | 관측 기간 |
|---|---|---|---|
| Explicit LE (응답 기반) | `dump_mongo_user.userinterestinfo` | 맞춤정보의 lifeevents 배열에 `_id` 값 1~6 응답 보유 | 최근 **3개월 내** 응답 (`updatedat` 기준) |
| Implicit LE (행동 기반) | `home_common.dm_user_le_implicit_da_d30` 마트 | 시공·이사·이사청소 신청일 보유 또는 가구 PDP 3개 이상 조회 | 최근 **30일** rolling |
| Cross-sell (시공) | `o2o.int_request`, `o2o.int_contract` | 시공 상담 신청 또는 시공 계약 체결 이력 보유 | **전체 기간 누적** |
| Cross-sell (이사) | `o2o.mov_request`, `o2o.mov_contract` | 이사 상담 신청 또는 이사 계약 체결 이력 보유 | **전체 기간 누적** |
| Cross-sell (인터넷) | **데이터 소스 미확인** | P7 산출 불가 사유 | — |
| M1/M2 행동 시그널 | `log.analyst_log_table` PAGEVIEW | 검색·PDP·리뷰 페이지 PV 합 vs 컨텐츠·스타일링샷 PV 합 비교 | 모수와 동일한 **7일** |

**왜 카테고리마다 관측 기간이 다른가:**

- Explicit LE (3개월): 맞춤정보 응답은 사용자가 자주 갱신하지 않습니다. 한 번 응답하면 한동안 유효한 정보로 활용 가능하기 때문에 3개월 정도의 비교적 긴 기간을 잡았습니다. 너무 짧게 (예: 7일) 잡으면 7일 안에 새로 응답한 유저만 잡혀서 Explicit 비율이 거의 0%에 가까워집니다.

- Implicit LE (30일 rolling): 마트 자체가 30일 rolling window로 집계되어 있어, 최근 한 달 안의 행동 시그널만 반영됩니다. 행동 데이터는 비교적 최신성이 중요해서 적절합니다.

- Cross-sell (전체 기간): 시공이나 이사 계약은 한 번 진행하면 그 유저는 영구적으로 "계약 경험이 있는 cohort"로 볼 수 있어, 시점 제한 없이 누적 데이터를 사용합니다.

- M1/M2 (7일): non LE 유저 안에서 M1 목적구매형 vs M2 맥락탐색형을 나누는 분류로, 모수와 동일한 7일을 봐야 정합성이 맞습니다. 모수에 없는 유저의 과거 행동을 끌어오면 안 되기 때문입니다.

### 1.3 LE 라이프이벤트 코드 (lifeevent ID 매핑)

`dump_mongo_user.userinterestinfo.lifeevents` 배열의 각 원소가 가지는 `_id` 값 매핑:

| `_id` | 라이프이벤트 의미 | 분류 활용 |
|:-:|---|---|

| **4** | 결혼 | P4 신혼부부 |
| **5** | 출산 | P8 출산, C 출산 |
| **6** | 주택건축 (단독주택 신축 등) | A 이사 LE에 흡수 |
| 7 | 콘텐츠 (영감 얻기 등) | LE 제외 (LE라기보다 일반 관심사) |
| 8 | 쇼핑 (그냥 둘러보기) | LE 제외 |

|:-:|---:|---|
| 1 | 211,057건 | P1 종합시공 |

**주의**: `remodelingtype` 값 1/2의 정확한 매핑 코드는 데이터 사전에 명시되어 있지 않아, 분포 기준으로 추정한 가정입니다. 사내 LE Squad에 확정 매핑 확인 필요.

### 1.5 Implicit LE (행동 기반) 시그널 상세 설명

`home_common.dm_user_le_implicit_da_d30` 마트는 "행동 기반으로 이사·시공 의향을 추정한 유저"를 담은 테이블입니다. 마트 이름 끝의 `d30`은 "최근 30일 rolling window"를 의미합니다.

**활용한 시그널 컬럼:**

| 컬럼명 | 의미 | LE 추정 |
|---|---|---|
| `construction_request_date` | 시공 상담 신청 날짜 (최근 신청일) | "시공/리모델링 진행 중이거나 진행 예정" → 시공 LE |
| `move_cleaning_request_date` | 이사청소 서비스 신청 날짜 (최근 신청일) | "이사 직전 또는 직후" → 이사 LE |
| `move_quote_request_date` | 이사 견적 신청 날짜 (최근 신청일) | "이사 예정" → 이사 LE |
| `furniture_unique_cnt` | 최근 30일 가구 PDP 조회 unique 상품 수 | 가구 탐색 행동량 (3개 이상이면 LE 진행 중으로 추정) |

**Implicit LE 판정 조건:**

위 컬럼 중 하나라도 다음을 만족하면 Implicit LE로 분류:
- `construction_request_date IS NOT NULL` (시공 상담 신청한 적 있음)
- 또는 `move_cleaning_request_date IS NOT NULL` (이사청소 신청한 적 있음)
- 또는 `move_quote_request_date IS NOT NULL` (이사 견적 신청한 적 있음)
- 또는 `furniture_unique_cnt >= 3` (가구 PDP 3개 이상 조회)

**중요: 실제 분류 결과에서 4가지 시그널 중 3가지는 실효성이 0임 (2.3 섹션 상세 참조).**
- Cross-sell(P5/P6)이 1순위라 시공·이사 신청 유저는 모두 그쪽으로 빠짐
- Implicit LE까지 내려오는 유저는 사실상 "가구 PDP 3개 이상" 조건만으로 분류된 유저

**Cross-sell(P5/P6)와의 차이:**

`construction_request_date`는 `o2o.int_request` 데이터와 거의 같은 정보이지만, 마트 형태로 30일 rolling 집계되어 있어 약간 다릅니다. 우선순위에서 P5(Cross-sell 시공)가 먼저 잡히므로, Cross-sell 보유 유저는 P5로 분류되고 Implicit으로는 빠지지 않습니다.

### 1.6 우선순위 (배타 분류 룰)

한 유저가 여러 카테고리에 동시에 해당할 수 있어, 우선순위를 정해 1개 카테고리로만 분류합니다.

**L3 그룹 분류 우선순위:**

```
1순위: B Cross-sell  (시공·이사 계약/상담 보유)
2순위: C 출산        (lifeevent 출산 응답)
3순위: A Explicit    (lifeevent 이사/리모델링/독립/결혼/주택건축 응답)
4순위: A Implicit    (시공·이사 신청 또는 가구 PDP 3개 이상)
5순위: non LE        (위 어디에도 해당 없음)
```

**L4 세그먼트 분류 우선순위 (시공 우선 원칙 적용):**

```
1순위: P5 시공 계약  (시공 상담/계약 보유)
2순위: P6 이사 계약  (이사 상담/계약 보유)
3순위: P1 종합시공   (lifeevent 리모델링 응답 + remodelingtype=1 or null)
4순위: P2 부분시공   (lifeevent 리모델링 응답 + remodelingtype=2)
5순위: P8 출산       (lifeevent 출산 응답, 이사 미응답)
6순위: P4 신혼부부   (lifeevent 결혼 응답, 이사 미응답)
7순위: P3 이사만     (lifeevent 이사 응답, 시공·결혼·출산 미응답)
8순위: Other Implicit LE (P1~P8 어디에도 안 잡히지만 행동 시그널 있음)
9순위: non LE        (위 어디에도 해당 없음)
```

**non LE 유저의 M1/M2 하위 분류:**

non LE 유저 안에서 행동 시그널로 두 모드를 구분합니다.

- M1 점수 = 검색·PDP·리뷰 페이지 PV 합 (최근 7일)
- M2 점수 = 컨텐츠·스타일링샷 페이지 PV 합 (최근 7일)

분류 룰:
- M1 점수 > M2 점수 → M1 목적 구매형
- M2 점수 > M1 점수 → M2 맥락 탐색형
- 둘 다 0 → Undetermined (홈만 보고 나간 유저)
- 동점 → M2로 분류 (오늘의집 특성상 맥락 탐색이 기본 모드라 가정)

### 1.7 "Other Implicit LE" 카테고리 정의

L4 분류 결과에 등장하는 카테고리로, 비중이 상당히 큽니다 (34.60%, 477,613명). 처음 보면 헷갈리는 분류명이라 별도로 정의를 풀어 씁니다.

**Other Implicit LE에 해당하는 유저 조건 (모두 만족해야 함):**

1. 시공·이사 상담/계약 이력 없음 (즉 P5, P6 아님)
2. 맞춤정보에 라이프이벤트 응답 없음 (즉 P1, P2, P3, P4, P8 아님)
3. 하지만 행동 시그널은 있음 (`dm_user_le_implicit_da_d30` 마트에 매핑되어 있음)
   - 즉, 시공·이사·이사청소 신청 이력 있음
   - 또는 최근 30일 가구 PDP 3개 이상 조회

**의미를 풀어 쓰면:**

이론적으로는 "LE 행동 시그널은 분명히 있는데, 어떤 LE인지 명시적으로 식별이 안 된 유저"입니다.

- 맞춤정보 응답을 안 했으니 "이사 LE인지 시공 LE인지 출산 LE인지" 알 수 없음
- 시공·이사 계약 단계까지 가지도 않았으니 Cross-sell도 아님
- 하지만 가구 PDP를 여러 개 보거나 (이론적으로는) 이사 견적을 신청하는 등 "LE스러운 행동"은 함

**실제로는 거의 100% "가구 PDP 헤비 탐색 유저" (중요)**

산출 결과 검증 시 발견된 사실:
- Other Implicit LE 477,603명 전원이 `furniture_unique_cnt >= 3` 조건으로만 분류됨
- 시공/이사/이사청소 신청 시그널 보유자: 0명 (모두 Cross-sell로 우선 흡수됨)

따라서 이 카테고리의 실제 의미는 **"최근 30일 가구 PDP 3개 이상 조회한 유저"** 한 그룹입니다. 이론적 정의보다 훨씬 단순합니다.

**왜 비율이 큰가 (34.60%):**

- 오늘의집 KR 유저의 핵심 행동이 가구 탐색·비교임을 반영
- 홈 → 추천 → PDP 1-tap 경로로 진입 장벽이 낮음
- 임계값 3은 "가벼운 둘러보기"와 "진지한 비교" 의 경계
- 가구 PDP는 LE와 무관하게 일상 쇼핑 행동으로도 발생하므로, 이를 "LE 시그널"로 봐도 되는지는 사내 정의 확인 필요

**L3 매핑:**

L3의 "A 이사 LE Implicit" (34.60%)과 거의 1:1로 대응됩니다. L3에서도 동일한 조건 (Cross-sell·출산·Explicit 없음 + Implicit만 있음)으로 분류하기 때문입니다.

## 2. 주의사항

### 2.1 데이터 정합성 관련

**모수는 unique 기준입니다**

7일 일별 PAGEVIEW를 단순 합산하는 방식과 다릅니다. 한 유저가 7일 중 5일 방문해도 1명으로 카운트합니다. 일별 합산 방식을 쓰면 헤비 유저가 중복 카운트되어 LE 비율이 부풀려질 수 있습니다.

**우선순위 배타 분류**

한 유저가 여러 카테고리 조건에 해당해도 우선순위 1순위 1개로만 분류됩니다. 예를 들어 시공 계약(P5)을 했으면서 이사 응답(P3)도 한 유저는 P5로만 잡힙니다. 덕분에 모든 카테고리 비율을 합하면 100%가 됩니다(중복 없음).

**각 카테고리 인원의 해석에 주의 (매우 중요)**

각 카테고리 유저 수는 "그 조건을 만족하는 유저 전체"가 아니라 "우선순위에 따라 그 카테고리로 분류된 유저"입니다. 

예시 1: B 신청/계약 (Cross-sell) 57,655명 (4.18%)
- 이건 "Cross-sell 행위가 있는 유저"이지 "Cross-sell만 하고 Explicit/Implicit은 안 한 유저"가 아닙니다
- 이 57,655명 중 일부는 동시에 lifeevent 응답(Explicit)이나 행동 시그널(Implicit)도 보유했을 수 있으나, Cross-sell 우선순위에 의해 B로만 잡혔습니다

예시 2: A 이사 LE Explicit 9.66%
- 이건 "Cross-sell 없으면서 Explicit 응답이 있는 유저"입니다
- "Explicit 응답이 있는 유저 전체"가 아닙니다 (Cross-sell 보유자는 B로 빠졌으므로)

"순수하게 X 카테고리만 해당하는 유저"의 비율이 궁금하면 우선순위 무관 조건 교집합으로 별도 산출해야 합니다.

**L3와 L4는 분류 기준이 달라 직접 비교 불가**

L3는 "어떤 LE 시그널 종류인가"로 분류하고, L4는 "어떤 라이프이벤트 페르소나인가"로 분류해서 우선순위 룰이 다릅니다.

| L3 (49.57%) | L4 매핑 관계 |
|---|---|
| B Cross-sell (4.18%) | P5 시공계약(3.14%) + P6 이사계약(1.03%) ≈ B 전체 |
| C 출산 (1.13%) | P8 출산(0.77%) + 일부는 P5/P6로 흡수 |
| A Explicit (9.66%) | P1(2.55%) + P2(0.67%) + P3(5.95%) + P4(0.85%) ≈ A Explicit 거의 전체 |
| A Implicit (34.60%) | Other Implicit LE (34.60%) |

검증: P1~P8 합 14.96% + Other Implicit LE 34.60% = 49.56% ≈ LE 전체 49.57% (반올림 오차)

흔한 오해: "A 이사 LE 44.26% = P1~P8 합" 이라고 생각하면 안 됨. A 안에는 P5/P6/P8이 안 들어있고 (이들은 L3에서 B/C로 분류됨), Other Implicit LE가 들어있음.

**로그인 유저만 포함**

비로그인 게스트 진입은 모수에서 제외했습니다. 비로그인까지 포함하려면 `bp_cid` (브라우저/디바이스 단위 ID) 기준으로 재산출이 필요합니다.

### 2.2 Explicit LE (응답 기반) 관련 주의

**3개월 응답 제한 적용**

`updatedat >= 2026-02-09` (산출 기준일 - 3개월) 조건을 걸어 1년 전 이사 응답 같은 만료된 응답을 제외했습니다. 사내 공식 만료 기준이 다를 경우 (예: 6개월, 이벤트 예정일 기준 등) 결과가 달라집니다.

**중복 응답 처리**

한 유저가 여러 라이프이벤트 (이사 + 리모델링 등)에 동시에 응답한 경우, 우선순위에 따라 1개 세그먼트로 분류됩니다. P1 종합시공 정의에 명시된 "시공 우선 원칙"이 적용되어, 시공+이사 동시 응답 시 P1으로 분류됩니다.

### 2.3 Implicit LE (행동 기반) 관련 주의

**Implicit LE 시그널의 실효성 (중요 발견)**

가이드 1.5 섹션에서 Implicit LE 판정 조건으로 4가지 시그널 (시공 상담 / 이사청소 / 이사견적 신청, 가구 PDP 3개 이상)을 명시했으나, **실제 분류 결과를 보면 4가지 중 3가지는 실효성이 사실상 0입니다.**

| 시그널 | Other Implicit LE 내 보유 유저 | 모수 대비 비율 |
|---|---:|---:|
| 시공 상담 신청 (`construction_request_date IS NOT NULL`) | 0명 | 0.00% |
| 이사청소 신청 (`move_cleaning_request_date IS NOT NULL`) | 0명 | 0.00% |
| 이사견적 신청 (`move_quote_request_date IS NOT NULL`) | 0명 | 0.00% |
| **가구 PDP 3개 이상 조회 (`furniture_unique_cnt >= 3`)** | **477,603명** | **34.60%** |

**왜 시공/이사 신청 시그널이 0인가:**

- 시공·이사·이사청소 신청 행동을 한 유저는 대부분 `o2o.int_request` 또는 `o2o.mov_request` 테이블에도 함께 기록됨
- 우선순위 룰에서 Cross-sell (B 카테고리, P5/P6)이 1순위라, 이 유저들은 Implicit 단계에 도달하기 전에 모두 Cross-sell로 흡수됨
- 결과적으로 Implicit LE 단계까지 내려오는 유저는 "신청 행동은 없으면서 가구만 탐색하는 유저"만 남음

**`furniture_unique_cnt` 마트 구조 특성:**

`home_common.dm_user_le_implicit_da_d30` 마트는 자체적으로 `furniture_unique_cnt >= 3` 필터를 적용해서 데이터를 담고 있는 것으로 보입니다. 확인 결과:
- 마트 내 `furniture_unique_cnt = 1, 2` 유저는 거의 없음
- 마트 내 유저 99.5%가 `furniture_unique_cnt >= 3`
- 즉, **마트 자체가 "가구 탐색 헤비 유저" 풀**

**Other Implicit LE 해석 재정립:**

처음 정의했던 "어떤 LE인지 식별 안 된 회색 영역 유저" 보다는, 실제로는 **"최근 30일 가구를 활발하게 둘러보고 있는 유저"** 한 그룹으로 해석하는 게 더 정확합니다.

- "이사 의향이 있는 유저" 라기보다 "가구를 사려고 비교 중인 유저"
- 이사 LE가 아니라 **구매 의향 (purchase intent) 지표**에 가까움
- 다만 가구 탐색이 이사·리모델링·결혼·출산 등 LE의 결과로 발생하기도 하므로, LE 시그널로 활용 자체는 유효

**실무 시사점:**

- 향후 Implicit LE 정의를 단순화 가능: 시공/이사 신청 시그널 (3가지)은 사실상 의미 없으므로 `furniture_unique_cnt >= 3` 단일 조건으로 충분
- 사내 LE Squad의 공식 "Implicit Grade 1" 정의가 실제로 어떤 시그널을 어떻게 결합하는지 확인 필요
- `furniture_unique_cnt` 임계값 (현재 3)을 5나 10으로 올리면 Implicit LE 비율이 줄어들면서 더 명확한 "구매 임박" 그룹을 식별 가능

**사내 공식 Implicit Grade 1 정의와 일치 여부 미검증**

현재 사용 중인 정의가 사내 LE Squad 공식 "Implicit Grade 1" 정의와 일치하는지 확인이 필요합니다. 특히:
- `furniture_unique_cnt` 임계값 3은 임의 설정 (사내는 다른 값일 수 있음)
- 사내 정의에는 PDP 외 다른 시그널 (검색, 스크랩 등)이 포함될 가능성

**Other Implicit LE 비중이 큰 이유 (34.60%)**

홈 진입 유저 중 1/3 이상이 "최근 30일 가구 PDP 3개 이상 조회" 그룹입니다. 이는:
- 오늘의집 KR 유저의 핵심 행동이 가구 탐색·비교임을 반영
- 가구 PDP는 진입 장벽이 낮아 (홈 → 추천 → PDP 1-tap) 일반 유저도 쉽게 도달
- 임계값 3은 "가벼운 둘러보기"와 "진지한 비교" 의 경계로 보이나, 더 좁히려면 임계값 상향 필요

### 2.4 Cross-sell 관련 주의

**P7 인터넷 상담/계약 데이터 미포함**

현재 시공(`o2o.int_*`)과 이사(`o2o.mov_*`)만 산출했고, 인터넷 계약 데이터는 별도 도메인에 있는 것으로 추정되어 산출하지 못했습니다. 사내 라이프서비스 팀이나 partners 도메인 어딘가에 있을 가능성이 높습니다.

**계약 시점 무관 누적**

`o2o.int_contract`, `o2o.mov_contract`는 전체 기간의 계약자를 다 포함합니다. 즉 5년 전에 계약한 유저도 포함됩니다. 만약 "최근 계약자만" 보고 싶으면 `request_date` 또는 `contract_inspected_date` 컬럼으로 추가 필터가 필요합니다.

### 2.5 P1 종합시공 vs P2 부분시공 분리 관련 주의

**`remodelingtype` 매핑이 가정 기반**

`remodelingtype = 1` → P1 종합시공, `remodelingtype = 2` → P2 부분시공 매핑은 데이터 분포 기준으로 추정한 가정입니다. 매핑 코드가 코드베이스에 명시되어 있지 않아 확정할 수 없었습니다.

- 분포: 1(211K) > null(204K) > 2(52K)
- 가장 많은 값을 종합시공으로, 적은 값을 부분시공으로 가정
- `null`은 응답값 미입력으로 보고 P1으로 흡수 (보수적 가정)

만약 매핑이 반대거나 (1=부분, 2=종합) null이 별도 의미라면 P1/P2 수치가 바뀝니다. 사내 LE Squad에 확정 매핑 확인이 필요합니다.

### 2.6 M1/M2 분류 관련 주의

**스크랩 액션 자체는 미포함**

스크랩(찜) 버튼 클릭 이벤트의 정확한 시그니처 (object_id, intent, action 조합)를 찾지 못해서, 대신 `MYPAGE_COLLECTION_BOOK` (스크랩북 페이지 진입)을 M2 시그널로 사용했습니다. 실제 "스크랩을 많이 누르는 유저"와는 차이가 있을 수 있습니다.

스크랩 액션 시그니처를 확보하면 M2 비율이 늘어날 가능성이 있습니다.

**Undetermined 9.21% 처리**

7일 안에 검색·PDP·리뷰·컨텐츠·스타일링샷 어느 것도 보지 않은 유저들입니다. non LE 안에서 가장 가벼운 유저층 (홈만 슬쩍 보고 이탈한 라이트 유저)으로 추정됩니다.

M1/M2만 따로 보면 모수가 달라져서 LE 비율과 안 맞습니다. 그래서 Undetermined를 별도 표시해 모수 정합성을 유지하는 게 권장됩니다.

### 2.7 기간 정합성 주의

**각 시그널의 관측 기간이 다름**

- 모수: 최근 7일 (2026-05-03 ~ 2026-05-09)
- M1/M2 행동 분류: 동일한 7일
- Implicit LE: 30일 rolling
- Explicit LE: 3개월 응답 제한
- Cross-sell: 전체 기간 누적

서로 다른 기간의 시그널을 모수 안에서 LEFT JOIN으로 매칭하는 구조라, 비교 시 이 차이를 염두에 둬야 합니다.

**1일치 vs 7일치 변동성 검증 결과**

7일 일별로 비율을 산출했을 때 모든 카테고리의 일별 변동성이 ±1.6%p 이내로 작게 나타났습니다. 즉 1일치 산출도 신뢰 가능하나, 보고용은 7일 평균 또는 7일 unique를 권장합니다.

### 2.8 변동 가능성이 큰 항목

비율이 정의 변경에 따라 크게 흔들릴 수 있는 항목을 정리합니다.

| 항목 | 현재 비율 | 변동 사유 |
|---|---:|---|
| Other Implicit LE | 34.60% | `furniture_unique_cnt` 임계값 (현재 3) 변경 시 비례 변동 |
| A Explicit | 9.66% | 응답 기간 제한 변경 시 (3개월 → 6개월 등) |
| P1 vs P2 | 2.55% vs 0.67% | `remodelingtype` 매핑 확정 시 |
| M1 vs M2 | 36.52% vs 4.70% | M2 시그널에 스크랩 액션 추가 시 M2 증가 |
| P7 인터넷 | 미산출 | 인터넷 계약 데이터 확보 시 추가 산출 가능 |

## 3. 재산출 가이드

### 3.1 다른 기간으로 재산출 시 수정 위치

쿼리 내 다음 4곳을 변경하면 다른 기간 분석 가능:

1. `home_users` CTE의 `WHERE date BETWEEN '시작일' AND '종료일'` 수정
2. `behavior_7d` CTE의 `WHERE date BETWEEN '시작일' AND '종료일'` 수정 (모수와 동일하게)
3. `explicit_moving`, `explicit_childbirth` CTE의 `INTERVAL '3' MONTH` 기준일 수정 (필요 시)
4. `implicit_le` CTE는 `MAX(base_dt)` 사용으로 자동 최신 반영

### 3.2 정의 변경 시 영향 범위

| 변경하는 정의 | 영향받는 카테고리 |

| Cross-sell 인터넷 추가 | P7 신설, B 증가, non LE 감소 |
| M1/M2 페이지 ID 추가/제거 | M1, M2, Undetermined 비율 모두 변동 |
| 우선순위 룰 변경 | 모든 P1~P8 비율 변동 |
